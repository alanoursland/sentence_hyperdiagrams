# Sentence Hyperdiagrams - Data Model

## Overview

A sentence hyperdiagram builds tree structures over a sequence of tokens by
attaching **labels** to tokens. Labels don't just annotate tokens — they link
other labels together, creating relationships between relationships.

Hyperdiagrams are constructed manually by human annotators, not generated by
AI or ML systems. The label ontology is hand-defined, drawing from classical
NLP and linguistics.

## Scope

The tree structure is not limited to syntax and grammar. Labels cover multiple
linguistic levels:

- **Morphology** — internal word structure
- **Syntax** — phrase structure, constituency, dependency
- **Semantics** — meaning, thematic roles, predicate-argument structure
- **Long-distance relationships** — coreference, binding, filler-gap
  dependencies, discourse relations

The same label mechanism spans all of these levels. Higher-level labels
(semantic, discourse) build on top of lower-level ones (syntactic) by
referencing them as children — relationships between relationships.

## Tokens

A token is the atomic unit of the input sequence (e.g., a word in a sentence).
Tokens can be considered labels without children — they are the leaf nodes of
any tree built from the label structure.

## Labels

A label is always attached to a single token. It creates a named tree node by
linking a label on the same token to a label on a previous token.

A label is a tuple:

    [name, child_prev, child_curr, index_prev, parameter, weight]

| Field        | Description                                              |
|--------------|----------------------------------------------------------|
| `name`       | Named identifier of the new tree node this label creates |
| `child_prev` | Named identifier of the label on the previous token      |
| `child_curr` | Named identifier of the label on the current token       |
| `index_prev` | Token index where `child_prev` is located                |
| `parameter`  | Optional 8-bit value (0-255) for ordinal variants of `name` |
| `weight`     | Confidence or strength of this label                     |

### Constraints

- A label is always attached to exactly one token.
- A label always links two children: one on the current token (`child_curr`)
  and one on a previous token (`child_prev` at position `index_prev`).
- `index_prev` < current token index — labels always point backward, so
  structure is built left to right.

### Tree construction

Each label produces a binary tree node:

```
        name
       /    \
  child_prev  child_curr
  (at index_prev)  (at current token)
```

Because labels reference other labels (not tokens directly), the tree is built
by chaining these pairwise links — forming relationships between relationships.

Multiple labels on a single token can participate in different parts of the
tree or build overlapping structures from the same token sequence.

The `parameter` field allows up to 256 ordinal variants of the same node name,
distinguishing otherwise identical node types.
